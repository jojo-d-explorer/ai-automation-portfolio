# Week 4 Reflection: Data Quality & Pipeline Standardization

*Week of February 23 – March 2, 2026*

## The Big Picture

Week 4 was about trust. Not building new features, but making the existing system reliable enough that I could trust it — and that the friends I'm serving could trust it.

The headline: **150+ dead or stale jobs purged from master databases across all 5 users.** That's not a bug fix. That's discovering the system had been confidently delivering garbage and building the infrastructure to prevent it.

## What I Built

### URL Verification System v2 (`check_urls.py`)

The original check_urls.py was naive — it just pinged URLs and looked for 404s. That caught maybe 30% of dead links. The rest showed HTTP 200 but were actually closed jobs.

**Platform-specific detection was the breakthrough:**

- **Ashby:** Uses client-side rendering (React). Standard HTTP requests see an empty HTML shell. Solution: Call their GraphQL API directly (`ApiJobPosting` query) to get definitive live/dead status.

- **Greenhouse/Lever:** Server-rendered, so content scanning works. Built a list of 23 closed-job phrases ("This position has been filled", "No longer accepting applications", etc.) and scan the page content.

- **JSON-LD metadata:** Many job pages embed structured data with `datePosted`. Extract this and flag anything >45 days old as STALE — even if the page is technically live, the job is probably filled.

- **Cross-domain redirects:** Dead ATS jobs often redirect to the company's generic careers page (different domain). Detect this pattern and mark as DEAD.

### LinkedIn Removal

This was an architecture decision, not a feature. Google's `site:linkedin.com/jobs` search returns cached pages from weeks or months ago. The jobs look active but are long closed.

**Solution:** Remove LinkedIn from automated search entirely. Friends receive personalized LinkedIn direct search URLs instead (generated by `linkedin_links.py`), which search LinkedIn directly rather than through Google's stale index.

This felt like giving up, but it was the right call. Platform-specific solutions beat generic approaches.

### 21-Column Schema

Evolved from 14 columns to 21. Key changes:
- **Dropped:** Language_Requirement (rarely populated, added noise)
- **Added:** Times_Seen, First_Seen_Date, Last_Seen_Date (persistence tracking)
- **Added:** Applied_Date, Application_Method, Response_Status, Interview_Stage, Notes (application tracking)

The schema is now standardized across all users. No more mid-pipeline schema upgrades breaking consolidation.

### 7-Phase Pipeline

What was implicit became explicit:

```
SEARCH → VERIFY → CLEAN → CONSOLIDATE → ANALYZE → PACKAGE → DELIVER
```

The key insight: **verification must happen before consolidation.** Dead links should never enter the master database. Moving check_urls.py from "sometime after search" to "Phase 2, immediately after search" was an architectural improvement that changed everything.

### ONE_CLICK v3.1

The v3.0 prompts were ambiguous about URL verification. CoWork would sometimes try to verify URLs during the search, which it can't actually do (no Python execution). This caused confusion.

**v3.1 is explicit:**
- Header: "This prompt does: SEARCH + SCORE only"
- `URL_Status` defaults to "Not Checked"
- Ends with: "STOP HERE. Run check_urls.py next."

Clear separation. Each phase does one thing.

## What I Learned

### 1. Trust erodes faster than it builds

One delivery with dead links — friends click and get 404s — destroys confidence in the entire system. A 33% false-positive rate meant 1 in 3 links was bad. That's not a minor issue; it's a credibility crisis.

**The fix wasn't optional polish.** Building check_urls.py v2 was foundational work that had to happen before any more deliveries.

### 2. Platform-specific behavior matters

There is no "generic URL check." Each platform has unique failure modes:
- Ashby: client-side rendering → need GraphQL
- LinkedIn: Google index stale → can't use programmatic search at all
- Greenhouse/Lever: server-rendered → content scanning works
- Cross-domain redirects: need to track URL changes through redirects

Generic solutions produce false confidence. You think you're checking URLs, but you're missing 70% of the dead ones.

### 3. Pipeline separation > monolithic prompts

The original ONE_CLICK tried to do everything: search, score, verify, deduplicate, format, save. When something went wrong, it was impossible to debug.

Breaking into discrete phases with clear handoffs:
- Makes each phase testable
- Makes failures localizable
- Makes the system understandable to future-me

This is basic software engineering. It just took me 4 weeks to learn it in the context of AI prompts.

### 4. Verification before consolidation (pipeline order matters)

Original flow: SEARCH → CONSOLIDATE → VERIFY

This meant dead links entered the master database, then had to be removed. The master database became polluted with historical garbage.

New flow: SEARCH → VERIFY → CONSOLIDATE

Dead links never enter master. The weekly CSV gets verified first; only survivors get consolidated. This seems obvious in retrospect, but it required rethinking the pipeline architecture.

### 5. Questioning arbitrary numbers is a feature

- Why 45 days for staleness? Because jobs typically fill in 30-60 days. 45 is a reasonable middle.
- Why 70-point score cutoff? Started at 80, found it was too restrictive. 70 captures "worth looking at" without overwhelming.
- Why 21 columns? Evolved from 14 based on what was actually useful.

None of these numbers came from a spec. They came from iteration. The willingness to question and adjust them is part of the skill.
